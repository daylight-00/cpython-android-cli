# CPython Android CLI + uv: Project Context

> **Status:** Active experiment
> **Current phase:** Stage 2-C synthesis
> **Primary target:** Termux on Android arm64 (`aarch64-linux-android`)
> **Host build environment:** Separate Linux workstation
> **Python baseline:** CPython 3.14.6
> **Last context update:** 2026-07-05

## 1. Purpose

This document is the current onboarding and handoff context for the project author, reviewers, and AI coding agents.

The project investigates how an upstream-built CPython Android runtime can be adapted into a normal CLI interpreter under Termux and used explicitly by uv.

The project is intentionally narrow. It is not currently trying to become:

- a CPython fork,
- a replacement for Termux Python,
- a uv-managed Python provider,
- a general Android distribution system,
- an Android port of `python-build-standalone`,
- a static Python distribution.

The working principle is:

```text
understand -> reproduce -> measure -> compare -> design -> optimize
```

not:

```text
patch -> patch more -> accidentally invent a distribution
```

## 2. Provenance

The current baseline lineage is:

```text
CPython 3.14.6 upstream source
        +
upstream CPython Android build infrastructure
        +
locally built aarch64-linux-android prefix
        +
minimal Android CLI frontend experiments
        +
Termux packaging and validation
        +
uv explicit interpreter selection
```

This wording matters.

The project did not begin from a generic Linux Python installation, and the runtime prefix used in the experiments should not be described as merely an unpacked official Python.org artifact. The relevant baseline is an upstream CPython Android build path and a locally built Android prefix.

A good project description is:

> **A CLI adaptation of an upstream CPython Android build for Termux, with uv integration.**

## 3. Responsibility boundaries

The project treats these layers separately:

```text
Python CLI semantics
    argv processing
    exit codes
    command/module/script modes

Python runtime initialization
    Py_BytesMain baseline
    PyConfig B0 selected frontend

Python path discovery
    sys.prefix
    sys.base_prefix
    sys.path
    venv identity

native dependency lookup
    libpython
    extension modules
    *_python.so dependency libraries

host trust-store integration
    SSL_CERT_FILE
    Termux CA bundle

external environment management
    uv explicit interpreter path
    uv venv
    uv pip
    uv run
```

A solution for one layer must not be assumed to solve the others.

## 4. Stage history

### Stage 1-A — explicit runtime baseline

Question:

> Can the Android CPython runtime work as a practical Termux CLI interpreter with uv when the runtime contract is made explicit?

Frontend:

```text
Py_BytesMain
```

Frozen runtime contract:

```text
LD_LIBRARY_PATH=<cpython-prefix>/lib
SSL_CERT_FILE=<termux-prefix>/etc/tls/cert.pem
```

Not required in the final Stage 1-A baseline:

```text
PYTHONHOME
PYTHONPATH
```

Validated areas included:

- basic CLI execution,
- native stdlib imports,
- HTTPS,
- uv venv,
- venv identity,
- native stdlib in the venv,
- uv pip install,
- package import,
- uv run.

Artifact-integrity validation used an external `PYTHONPYCACHEPREFIX` so generated bytecode did not mutate the extracted runtime prefix.

See `docs/stages/STAGE1A_BASELINE.md`.

### Stage 1-B — PyConfig frontend comparison

Question:

> Can an explicit PyConfig-based frontend preserve the validated Stage 1-A behavior?

Variants:

```text
Stage 1-A
    Py_BytesMain baseline

B0
    PyConfig_InitPythonConfig
    PyConfig_SetBytesArgv
    Py_InitializeFromConfig
    PyConfig_Clear
    Py_RunMain
    no config.home override

B1
    same PyConfig flow
    config.home set from CPYTHON_HOME
```

Broad functional and semantic differential testing found no regression for B0 on the tested surface.

B1 also worked with the correct home, but a wrong-home negative control failed during startup because Python could not import `encodings`. This proved that the explicit home value was a real additional dependency.

Decision:

```text
selected:
  B0 / PyConfig auto-discovery

not selected:
  B1 / explicit config.home
```

The Stage 1-A external runtime contract remained unchanged.

See `docs/stages/STAGE1B_PYCONFIG.md`.

### Stage 2-A — bootstrap strategy comparison

Starting point:

```text
B0 PyConfig frontend
+
external LD_LIBRARY_PATH
+
external SSL_CERT_FILE
```

Three strategies were compared.

#### S2-E: setenv-only

Result:

```text
current process native imports: FAIL
subprocess inheriting prepared environment: PASS
```

Interpretation:

Updating the process environment was useful for newly executed processes but did not repair native dependency loading in the already-running process.

#### S2-U: bionic linker update experiment

The experiment attempted to resolve `android_update_LD_LIBRARY_PATH` through `RTLD_DEFAULT`.

Result on the tested target:

```text
symbol unavailable through the tested resolution path
```

This strategy was not selected.

#### S2-R: self re-exec

The launcher prepared the loader path and CA environment, then re-executed itself so the fresh process started with the required native search path.

Result:

```text
native stdlib: PASS
HTTPS: PASS
subprocess: PASS
uv venv: PASS
venv native imports: PASS
uv run: PASS
```

S2-R was selected for refinement.

### Stage 2-B — R2 conditional re-exec

Stage 2-B removed the private recursion guard and used the environment state itself as the fixed point.

Policy:

```text
resolve /proc/self/exe
        |
derive prefix/lib
        |
normalize LD_LIBRARY_PATH
        |
        +-- required libdir absent at process start
        |       configure CA
        |       execv(actual executable, original argv)
        |
        +-- required libdir already present
                repair/discover CA if needed
                B0 PyConfig
                Py_RunMain
```

Validated properties:

- clean launch performs `reexec -> direct`,
- ready launch is direct-only,
- wrong native path is repaired,
- wrong CA path is repaired independently,
- duplicate required path entries are normalized,
- unrelated cwd works,
- external symlink invocation works,
- subprocess child avoids redundant re-exec,
- native stdlib imports work,
- HTTPS works,
- uv venv works,
- clean venv launch works,
- venv identity is preserved,
- uv run works.

Whole-prefix relocation was also validated:

```text
location A: PASS
move entire prefix A -> B
location B: PASS
```

The relocation test covered the base runtime, native imports, HTTPS, subprocess re-entry, new uv venv creation, clean venv launch, and uv run.

Boundary:

```text
whole runtime prefix relocation:
  tested and passed

previously-created external venv after base-runtime move:
  not tested
```

## 5. Current selected architecture

```text
shell / uv / venv entry point
        |
        v
Stage 2-C launcher
        |
        +-- read /proc/self/exe
        +-- derive actual prefix
        +-- derive <prefix>/lib
        +-- normalize LD_LIBRARY_PATH
        +-- preserve a valid SSL_CERT_FILE
        +-- otherwise discover Termux CA bundle
        |
        +-- required libdir was missing at process start
        |       -> execv(actual executable, original argv)
        |
        +-- required libdir was already present
                -> B0 PyConfig initialization
                -> Py_RunMain
```

The launcher-level build uses a relative RUNPATH for direct `libpython3.14.so` lookup:

```text
$ORIGIN/../lib
```

This should not be confused with the broader extension dependency problem that Stage 2 solves through bootstrap and re-exec.

## 6. uv's role

uv is a consumer of the interpreter in this project, not the builder of the interpreter.

The supported integration model is explicit interpreter selection, for example:

```sh
uv venv \
  --no-python-downloads \
  --python /absolute/path/to/python \
  .venv
```

The project does not currently need to imitate uv-managed Python archive metadata or provide automatic uv Python downloads.

## 10. Stage 2-C objective

Stage 2-C is not selecting a new bootstrap mechanism.

Its job is to prove that the validated R2 architecture survives the cleaned project workflow:

```text
workstation source
    -> canonical build script
    -> canonical out artifact
    -> artifact sync
    -> Termux runtime assembly
    -> production-shape smoke validation
```

Completion requires:

- clean base runtime execution,
- native stdlib imports,
- HTTPS,
- subprocess re-entry,
- uv venv,
- clean venv execution,
- venv identity,
- uv run.

After this end-to-end workflow passes, Stage 2 can be frozen.

## 11. Non-goals before Stage 3

Do not prematurely add:

- mass ELF patching,
- private Android linker internals,
- static CPython linking as a requirement,
- custom full dependency rebuild infrastructure,
- uv-managed automatic Python downloads,
- `python-build-standalone` architecture copied wholesale.

Those may become later research topics, but they should not be mixed into Stage 2-C without evidence that the current architecture requires them.

## 12. Reference hierarchy

Use references in this order:

1. CPython upstream Android documentation and build infrastructure.
2. CPython CLI and initialization APIs (`Programs/python.c`, PyConfig, `Py_RunMain`).
3. Astral uv documentation and Android integration work for explicit interpreter workflows.
4. BeeWare CPython Android source-dependency work for native dependency investigation.
5. `python-build-standalone` for later distribution-engineering questions.

## 13. Handoff rules

Before changing architecture:

1. identify which layer the change belongs to;
2. state what validated behavior it must preserve;
3. compare against the selected R2 + B0 baseline;
4. avoid changing Python initialization, native loader strategy, CA trust integration, and packaging structure in one uncontrolled experiment;
5. keep evidence-bearing failed variants under `experiments/` when they explain a decision;
6. treat `config/legacy/` as historical input, not active configuration;
7. keep build outputs under the canonical `out/<target>/<profile>/` tree;
8. keep workstation and Termux absolute paths out of tracked current scripts.

The next task is:

> Run the complete Stage 2-C workstation-build -> artifact-sync -> Termux-assembly -> smoke-validation workflow from the cleaned repository structure, then freeze Stage 2 if the behavior matches the Stage 2-B baseline.
